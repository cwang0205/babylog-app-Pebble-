BabyLog Technical Documentation

Project Status: Production Ready (Version 2.0)
Core Value: Voice-first, AI-powered tracking with real-time cloud synchronization and family sharing.

1. System Architecture

The application follows a Serverless / Thick-Client architecture. The React frontend handles all logic, audio recording, and direct communication with external APIs (Firebase & Google Gemini).

Layer Technology Role
Frontend React 19, TypeScript, Tailwind CSS UI rendering, State Management, Audio Capture (MediaRecorder API).
Auth & DB Firebase Auth, Cloud Firestore Identity management (Google Sign-In) and NoSQL document storage.
Intelligence Google Gemini API (gemini-3-flash-preview) Multimodal processing. Converts raw audio/text into structured JSON.
Hosting Static Web Host (e.g., Vercel/Firebase Hosting) Delivers the SPA bundle.

2. Core Process Flows

A. The "Magic" Logging Flow (Voice Input)

This is the primary differentiator of the product. It allows hands-free logging.

1
User Input: User taps the microphone button and speaks natural language (e.g., "Leo just drank 4oz of milk"). Component: VoiceRecorder.tsx | API: navigator.mediaDevices.getUserMedia
2
Capture & Encoding: Audio is captured as chunks and compiled into a Blob (WebM or MP4). Format: audio/webm or audio/mp4
3
AI Processing: The Blob is converted to Base64 and sent to Gemini.
Prompt Injection: The system injects `Current User Time` into the prompt so the AI can resolve relative time (e.g., "10 minutes ago"). Service: GeminiService.parseInput | Model: gemini-3-flash-preview
4
Structured Output: Gemini returns a JSON object adhering to the schema defined in `geminiService.ts`. Returns: ParseResult (Type, StartTime, Details, Notes)
5
Human-in-the-loop Review: A modal appears allowing the user to correct the AI's inference before saving. Component: EventConfirmation.tsx
6
Persistence: Data is written to Firestore `events` collection. Service: StorageService.addEvent

B. Family Sharing & Authorization Flow

Enables multiple caregivers (parents, nannies, grandparents) to track the same baby.

1
Invitation: Owner clicks "Share", enters an email address (e.g., `spouse@gmail.com`). Component: ShareBabyModal.tsx
2
Database Update: The email is added to the `allowedEmails` array on the Baby Profile document. Firestore Operation: arrayUnion('spouse@gmail.com')
3
Security Check (Rule Evaluation): When the invitee logs in, Firestore Security Rules check if `request.auth.token.email` exists in the `resource.data.allowedEmails` list. File: firestore.rules
4
Sync: The invited user's dashboard automatically queries for all babies where their email is listed. Query: where("allowedEmails", "array-contains", userEmail)

3. Data Dictionary (Firestore Schema)

Collection: babies

Field Type Description
id String (UUID) Document ID.
name String Baby's display name.
ownerId String (UID) Firebase Auth UID of the creator. Has delete permissions.
allowedEmails Array<String> List of emails permitted to Read/Write events for this baby.

Collection: events

Field Type Description
babyId String (Ref) Links event to parent Baby profile.
type String (Enum) feed, sleep, diaper, symptom, measurement, note.
startTime ISO String When the event occurred.
endTime ISO String Optional. Used for Sleep durations.
details Map (JSON) Flexible payload.
E.g., { amountml: 120, side: "left" } for feeds.
createdByEmail String Audit trail of who logged the specific event.

4. Security & Permissions

Access control is enforced at the database level using Firestore Security Rules.

  • Read/Write: Granted if User UID matches `ownerId` OR User Email is in `allowedEmails`.
  • Delete Profile: Granted ONLY if User UID matches `ownerId`.
  • Create Profile: Authenticated users only.

6. Recent Technical Updates & Optimizations

A. Mobile Date Picker Reliability

Fixed an issue where the native date picker would not consistently open on mobile browsers (specifically iOS Safari).

  • Fix: The invisible <input type="date"> was forced to stretch to the exact dimensions of its container (w-full h-full) and pulled to the front layer (z-10).
  • Fallback: Added an onClick handler that explicitly calls showPicker() on the input element to force the native UI to open.

B. 30-Day Health Log (Optimized Querying)

The Health Log (Symptoms and Measurements) now operates on a separate, optimized 30-day timeline independent of the main 14-day dashboard view.

1
Independent Subscription: A dedicated onSnapshot listener fetches only symptom and measurement events for the last 30 days. Service: StorageService.subscribeToHealthEvents
2
Read Optimization: This query runs exactly once when the app loads or the active baby changes. It does not re-run when the user navigates through different dates on the main timeline, saving significant Firestore document reads.
3
Required Firestore Index (Manual Step): Because this query filters by both type and startTime, Firestore requires a composite index.

Action Required: You must manually create this index in the Firebase Console. The app will log the exact creation URL in the browser console if the index is missing.
Index Fields: babyId (Ascending), type (Ascending), startTime (Ascending).

C. Overnight Sleep Splitting (Analysis Engine)

The ConsolidatedReport engine was rewritten to accurately calculate sleep durations that cross the midnight boundary.

  • Previous Behavior: A sleep from 8 PM to 6 AM was credited entirely (10 hours) to the day it started.
  • New Behavior: The engine calculates the exact overlap of the sleep session with the calendar day.
    • Day 1 receives 4 hours (8 PM to Midnight).
    • Day 2 receives 6 hours (Midnight to 6 AM).
  • Sleep Count: The total number of sleep sessions (count) is still credited to the day the sleep initiated.

7. Offline Capabilities & Capacitor Integration

To support iOS App Store requirements and ensure graceful degradation, the app utilizes Firestore's built-in offline persistence. This allows the React app (running inside a Capacitor WebView) to function when the device loses network connectivity.

A. Core Offline Flow

1
Initialization: Firestore is initialized with persistentLocalCache. This stores recently loaded documents in the device's local IndexedDB. Service: firebase.ts | API: initializeFirestore(app, { localCache: persistentLocalCache(...) })
2
Offline Data Flow (Reads): When the user opens the app without a network connection, the existing onSnapshot listeners automatically serve data from the local IndexedDB cache. The UI renders the last-known state seamlessly.
3
Offline Data Flow (Writes): If the user manually logs an event (via the manual entry forms), Firestore queues the write operation locally. The UI updates immediately (optimistic UI), and the SDK automatically syncs the queued writes to the cloud once the connection is restored.
4
Graceful Degradation (AI Logging): Because the Gemini LLM API requires a network connection, voice AI logging is disabled while offline. The app listens to the browser's online and offline events to display a sticky warning banner and block AI processing attempts with a clear alert. Component: App.tsx | State: isOffline

B. Advanced Sync & Cache Behavior

  • Multi-Device Sync (e.g., Co-parenting): The onSnapshot listener acts as a highly intelligent traffic controller. It immediately grabs data from the Local Cache so the screen loads instantly, then connects to the Cloud to check for any changes. If multiple users (e.g., Mom and Dad) are online, a change made by one is instantly pushed to the other's listener.
  • Conflict Resolution (Last Write Wins): If two offline devices edit the exact same event (e.g., Mom changes a diaper log to "Wet", Dad changes it to "Dirty"), the device that reconnects to the internet last will overwrite the other based on server timestamps. If they create new independent events offline, both are safely added to the timeline upon reconnection without glitches.
  • Delta Syncing (Mutation Queue): When offline, Firestore does not save the final state of the database; it creates a hidden queue of specific mutations (e.g., Action 1: Added feeding, Action 2: Deleted sleep). Upon reconnection, it silently replays only that specific queue of actions to the cloud in a fraction of a second. It does not re-upload the entire dataset, saving bandwidth and battery.
  • Cache Limits & Garbage Collection: The default local cache size is 40 MB, which can hold years of text-based JSON baby logs. If the cache ever exceeds this limit, Firestore uses a Least Recently Used (LRU) garbage collector to silently remove the oldest, least-viewed data from the local device's memory to make room for new data (this does not delete data from the cloud).
